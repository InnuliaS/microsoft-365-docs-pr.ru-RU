# <a name="create-dns-records-at-any-dns-hosting-provider"></a>Создание записей DNS у любого поставщика услуг размещения DNS

 Если вы не нашли то, что вы ищете, см. раздел **[Вопросы и ответы по доменам](../setup/domains-faq.md)**. 
  
Просмотрите наш список [инструкций для конкретных хостов](https://support.office.com/article/ae950c9e-e8d9-4108-b0cb-449156998580). Найдите в нем свой хост и следуйте указаниям, чтобы добавить все нужные записи. 
  
Если вы не знаете, какой поставщик услуг размещения DNS или регистратор используется для домена, см. статью [Определение регистратора домена или поставщика услуг размещения DNS](../get-help-with-domains/find-your-domain-registrar.md).    
  
При самостоятельной настройке добавляются указанные ниже записи. Обратите внимание, что проверочная запись и запись MX уникальны для вашего домена. Чтобы их настроить, вам потребуется получить и использовать особое значение "маркера" для вашего домена. Ниже описано, как это сделать.
  
> [!IMPORTANT]
> Точные названия областей или *полей*, в которые вы вводите или вставляете данные для создания разных типов DNS-записей, различаются в зависимости узла DNS. На веб-сайте вашего узла DNS может находиться справка для сопоставления приведенных здесь инструкций с определенными полями веб-сайта. Не забудьте проверить наличие пошаговых инструкций для вашего поставщика услуг DNS в разделе [Создание записей DNS для Microsoft 365](https://support.office.com/article/b0f3fdca-8a80-4e8e-9ef3-61e8a2a9ab23.aspx).  > Некоторые узлы DNS не позволяют создать все записи требуемых типов, из-за чего [ограничиваются возможности обслуживания](https://support.office.com/article/7ae9a655-041d-4724-aa92-60392ee390c2.aspx) в Microsoft 365. Если узел вашего домена не поддерживает, например, записи SRV, TXT, или CNAME, мы рекомендуем [перенести домен](../get-help-with-domains/buy-a-domain-name.md) на узел DNS, поддерживающий все необходимые записи. Чтобы воспользоваться быстрым автоматическим процессом настройки для Microsoft 365, рекомендуем перенести домен в службу GoDaddy. 
  
> [!NOTE]
> Обычно изменения DNS вступают в силу всего через несколько минут. Тем не менее, иногда распространение внесенного изменения в системе DNS по всему Интернету занимает больше времени. Если после добавления записей DNS возникла проблема с потоком обработки почты или другие неполадки, см. статью [Поиск и устранение неполадок после смены имени домена или записей DNS](../get-help-with-domains/find-and-fix-issues.md). 
  
## <a name="add-a-txt-or-mx-record-for-verification"></a>Добавьте запись TXT или MX для проверки
<a name="BKMK_verify"> </a>

> [!NOTE]
> Нужно создать только одну из этих записей. TXT является предпочтительным типом, но некоторые поставщики услуг размещения DNS не поддерживают его. В таком случае можно создать MX-запись. 
  
Прежде чем вы сможете использовать свой домен в Microsoft 365, мы должны убедиться в том, что вы являетесь его владельцем. Если вы войдете в свою учетную запись на сайте регистратора доменных имен и создадите запись DNS, это послужит для Microsoft 365 подтверждением того, что вы владеете доменом.
  
> [!NOTE]
> Эта запись используется исключительно для проверки принадлежности домена. При желании вы сможете удалить ее позже. 
  
 **Найдите раздел для создания записей на веб-сайте поставщика услуг размещения DNS.**
  
1. Войдите на сайт поставщика услуг размещения DNS.
    
2.  Выберите свой домен.
    
3. Найдите страницу, на которой можно редактировать DNS-записи для домена.
    
 **Создайте запись.**
  
1. При создании записи TXT или MX следуйте соответствующим инструкциям.
    
   - **Для создания TXT-записи используйте следующие значения:**
    
      |||||
      |:-----|:-----|:-----|:-----|
      |**Record Type** (Тип записи)|**Alias** (Псевдоним) или **Host Name** (Имя узла)|**Value** (Значение)|**TTL** (Срок жизни)|
      |TXT|Выполните одно из следующих действий: Введите **@**, оставьте поле пустым или укажите имя своего домена.  <br/> **Примечание.** У различных узлов DNS могут быть разные требования к этому полю. |MS=ms *XXXXXXXX*  <br/> **Примечание.** Это пример. Используйте здесь собственное значение **Назначение или адрес "указывает на"** из таблицы в Microsoft 365.  <br/>        [Как его найти?](../get-help-with-domains/information-for-dns-records.md)     <br/>     |Задайте для этого значения **1 hour** (1 час) или эквивалент в минутах ( **60** ), секундах ( **3600** ) и т. д.  |
   
   - **Для создания MX-записи используйте следующие значения:**
    
      ||||||
      |:-----|:-----|:-----|:-----|:-----|
      |**Record Type** (Тип записи)|**Alias** (Псевдоним) или **Host Name** (Имя узла)|**Value** (Значение)|**Priority** (Приоритет)|**TTL** (Срок жизни)|
      |MX|Введите знак **@** или имя своего домена. |MS=ms *XXXXXXXX* <br/> **Примечание.** Это пример. Используйте здесь собственное значение **Назначение или адрес "указывает на"** из таблицы в Office 365.    <br/>       [Как его найти?](../get-help-with-domains/information-for-dns-records.md)     <br/>     |В поле **Priority** (Приоритет) используйте более низкое значение приоритета, чем указанное для других записей MX, во избежание конфликтов с записью MX, используемой для потока обработки почты. <br/> Дополнительные сведения о приоритете см. в статье [Что такое приоритет записей MX?](https://support.office.com/article/2784cc4d-95be-443d-b5f7-bb5dd867ba83.aspx). <br/> |Задайте для этого значения **1 hour** (1 час) или эквивалент в минутах ( **60** ), секундах ( **3600** ) и т. д. |
   
2. Сохраните запись.
    
Теперь, когда запись добавлена на сайте регистратора доменных имен, вернитесь в Microsoft 365 и подайте запрос на ее поиск.
  
Если Microsoft 365 обнаружит правильную запись TXT, это значит, что домен проверен.
  
1. В Центре администрирования перейдите на страницу **Settings** (Параметры) \> <a href="https://go.microsoft.com/fwlink/p/?linkid=834818" target="_blank">Domains</a> (Домены).
    
2. На странице **Domains** (Домены) выберите домен, который нужно проверить. 
  
3. На странице **Setup** (Настройка) выберите **Start setup** (Начать настройку).
       
4. На странице **Проверка домена** выберите **Проверить**.   
  
> [!NOTE]
>  Обычно на вступление изменений DNS в силу требуется около 15 минут. Однако иногда распространение внесенного изменения в системе DNS по всему Интернету занимает больше времени. Если после добавления записей DNS возникла проблема с потоком обработки почты или другие неполадки, см. статью [Устранение неполадок после смены имени домена или записей DNS](../get-help-with-domains/find-and-fix-issues.md). 
  
## <a name="add-mx-record-to-route-email"></a>Добавление записи MX для маршрутизации электронной почты
<a name="BKMK_add_MX"> </a>

Добавьте запись MX, чтобы сообщения электронной почты для вашего домена доставлялись в Microsoft 365.  *После обновления записи MX вашего домена все новые сообщения, адресованные его пользователям, будут поступать в Microsoft 365*. Ранее полученные сообщения останутся на текущем узле электронной почты, если вы не решите [перенести сообщения и контакты в Microsoft 365](../setup/migrate-email-and-contacts-admin.md).

 **Задача**
  
Найдите страницу, на которой можно создавать записи для вашего домена.
  
1. Войдите на веб-сайт вашего узла DNS.
    
2.  Выберите свой домен.
    
3. Найдите страницу, на которой можно редактировать DNS-записи для домена.
    
::: moniker range="o365-worldwide"

  Добавляемая запись MX включает значение **Points to address** (Адрес "указывает на") следующего вида: \<маркер MX\>.mail.protection.outlook.com, где \<маркер MX\>  это значение в формате MSxxxxxxx. 

::: moniker-end

::: moniker range="o365-germany"

  Добавляемая запись MX включает значение **Points to address** (Адрес "указывает на") следующего вида: \<маркер MX\>.mail.protection.outlook.de, где \<маркер MX\> — это значение в формате MSxxxxxxx. 

::: moniker-end

4. На веб-сайте узла DNS щелкните команду для добавления новой записи MX.
    
    Теперь вы [получите информацию для записи MX](../get-help-with-domains/information-for-dns-records.md) из Microsoft 365. 
    
5. Для записи MX (на указанном выше шаге) скопируйте значение **Points to address** (Адрес "указывает на"). 
    
    Это значение будет использоваться в записи, создаваемой на сайте DNS-узла, как описано в следующем действии.
    
6. Убедитесь, что значения полей новой записи MX на сайте DNS-узла точно соответствуют указанным ниже.
    
   - **Record Type** (Тип записи): **MX**
    
   - **Priority** (Приоритет): задайте для приоритета MX-записи наивысшее из доступных значений (как правило, это **0**).
    
      Дополнительные сведения о приоритете см. в статье [Что такое приоритет записей MX?](https://support.office.com/article/2784cc4d-95be-443d-b5f7-bb5dd867ba83.aspx).
    
   - **Host Name** (Имя узла): **@**
    
   - **Points to address** (Адрес "указывает на"): вставьте значение поля **Points to address**, скопированное из Microsoft 365. 
    
   - **TTL** (Срок жизни): задайте значение **1 час** или эквивалент в минутах (**60**), в секундах (**3600**) и т. д. 
    
7. Сохраните запись.
    
Удалите все остальные записи MX.
  
Если есть записи MX для этого домена, направляющие электронную почту на другой адрес (не в Microsoft 365), удалите их все.

## <a name="add-a-txt-record-for-spf-to-help-prevent-email-spam"></a>Добавьте запись TXT для SPF, чтобы предотвратить рассылку спама
<a name="BKMK_add_MX"> </a>

::: moniker range="o365-worldwide"

> [!IMPORTANT]
> Для записи инфраструктуры политики отправителей (SPF) для домена можно указать только одну запись TXT. Если у вашего домена больше одной записи SPF, это приведет к сбоям в работе почты и ошибкам классификации входящих писем и спама. Если вы уже указали запись SPF для домена, не создавайте еще одну для Microsoft 365. Вместо этого добавьте необходимые значения Microsoft 365 в текущую запись. Таким образом, в *одной* записи SPF будут указаны оба набора значений.
  
На веб-сайте своего узла DNS измените имеющуюся запись SPF или создайте новую запись TXT для SPF.
  
> [!IMPORTANT]
> Инфраструктура политики отправителей (SPF) призвана предотвратить спуфинг, но существуют некоторые методики, позволяющие обойти ее. Чтобы защититься от таких атак, по завершении настройки SPF необходимо настроить DKIM и DMARC для Microsoft 365. Чтобы начать, см. статью [Проверка исходящей электронной почты, отправляемой со своего домена в Microsoft 365, с помощью DKIM](https://technet.microsoft.com/library/mt695945%28v=exchg.150%29.aspx). Затем см. статью [Использование протокола DMARC для проверки электронной почты в Microsoft 365](https://technet.microsoft.com/library/mt734386%28v=exchg.150%29.aspx). 
  
1. В полях новой записи введите (или скопируйте и вставьте) приведенный ниже набор значений, который подходит для вашей ситуации.
    
    |||||
    |:-----|:-----|:-----|:-----|

## <a name="add-two-srv-records"></a>Добавьте две записи SRV
<a name="BKMK_add_MX"> </a>

::: moniker range="o365-worldwide"

На веб-сайте узла DNS вы создадите две новые записи SRV, обычно по одной за раз. Другими словами, после добавления первой записи SRV на веб-сайте выберите создание еще одной записи.
  
1. В поля для каждой новой записи введите (или скопируйте и вставьте) указанные ниже значения. **(Инструкции по созданию записей SRV, когда на узле DNS нет отдельных полей для ввода всех значений, см. ниже.)**
    
    ||||||||||
    |:-----|:-----|:-----|:-----|:-----|:-----|:-----|:-----|:-----|
    |**Record Type** (Тип записи) <br/> |**Name** (Имя) <br/> |**Target** (Целевое значение) <br/> |**Protocol** (Протокол) <br/> |**Service** (Служба) <br/> |**Priority** (Приоритет) <br/> |**Weight** (Вес) <br/> |**Port** (Порт) <br/> |**TTL** (Срок жизни) <br/> |
    |SRV (Service)  <br/> |@  <br/> (Или оставьте это поле пустым, если значение @ не разрешено.)  <br/> |sipdir.online.lync.com  <br/> |_tls  <br/> |_sip  <br/> |100  <br/> |1  <br/> |443  <br/> |1 час  <br/> |
    |SRV (Service)  <br/> |@  <br/> (Или оставьте это поле пустым, если значение @ не разрешено.)  <br/> |sipfed.online.lync.com  <br/> |_tcp  <br/> |_sipfederationtls  <br/> |100  <br/> |1  <br/> |5061  <br/> |1 час  <br/> |
   
    > [!NOTE]
    >  Оставьте поле **Name** (Имя) пустым, если узел DNS не позволяет задать для него значение **@**. Следуйте этим инструкциям *только* в том случае, если на узле DNS предусмотрены отдельные поля для значений "Service" (Служба) и "Protocol" (Протокол). В противном случае см. примечания о соответствующих параметрах ниже. 
    > 
    >  Для значений **Service** (Служба) и **Protocol** (Протокол): если у вашего узла DNS нет этих полей для записей SRV, необходимо указать значения **Service** и **Protocol** в качестве значения параметра **Name** (Имя) записи. (Примечание. В зависимости от узла DNS, поле **Name** может называться иначе, например **Host**, **Hostname** или **Subdomain**.) Чтобы задать несколько параметров в одном значении, укажите из в одной строке, разделив значения точкой.  Пример: **Name** = _sip._tls 
    > 
    >  Для значений **Priority** (Приоритет), **Weight** (Вес) и **Port** (Порт): если на узле DNS нет необходимых полей для записей SRV, необходимо задать эти параметры в значении параметра **Target** (Цель) этой записи. (Примечание. В зависимости от узла DNS, поле **Target** (Цель) может называться иначе, например **Content**, **IP Address** или **Target Host**.) Чтобы задать несколько параметров в одном значении, создайте одну строку, в которой значения должны быть разделены пробелами. Эта строка должна оканчиваться точкой. Значения указываются в следующем порядке: Priority, Weight, Port, Target. Пример: **Target**: 100 1 443 sipdir.online.lync.com. 
    > 
    >  Варианты для значений **Priority** (Приоритет), **Weight** (Вес) и **Port** (Порт): на некоторых узлах DNS некоторые (но не все) поля могут предоставляться отдельно. На сайтах таких узлов DNS укажите значения, для которых не предусмотрены отдельные поля, в одной строке в нужном порядке в значении параметра **Target** (Цель). (Примечание. В зависимости от узла DNS, поле **Target** (Цель) может называться иначе, например **Content**, **IP Address** или **Target Host**.) Чтобы задать несколько параметров в одном значении, создайте одну строку для параметров, для которых не предусмотрены отдельные поля. Значения этих параметров должны быть разделены пробелами. Значения задаются *в следующем порядке* (параметры, для которых предусмотрены отдельные поля, опускаются): Priority, Weight, Port, Target. Например, если для параметра Priority существует отдельное поле, в одну строку нужно объединить только значения Weight, Port и Target: **Target**: 1 443 sipdir.online.lync.com 
    > 
    > Для поля **TTL** (Срок жизни): установите значение **1 час** или эквивалент в минутах (**60**), в секундах (**3600**) и т. д. 
  
2. По завершении сохраните записи.
    
    > [!NOTE]
    >  Обычно на вступление изменений DNS в силу требуется около 15 минут. Однако иногда распространение внесенного изменения в системе DNS по всему Интернету занимает больше времени. Если после добавления записей DNS возникла проблема с потоком обработки почты или другие неполадки, см. статью [Устранение неполадок после смены имени домена или записей DNS](../get-help-with-domains/find-and-fix-issues.md).

Значения задаются *в следующем порядке* (параметры, для которых предусмотрены отдельные поля, опускаются): Priority, Weight, Port, Target. > Например, если для параметра Priority существует отдельное поле, в одну строку нужно объединить только значения Weight, Port и Target: **Target**: 1 443 sipdir.online.partner.lync.cn 
    > 
    >  Для поля **TTL** (Срок жизни): установите значение **1 час** или эквивалент в минутах (**60**), в секундах (**3600**) и т. д. 
  
2. По завершении сохраните записи.
    
    > [!NOTE]
    >  Обычно на вступление изменений DNS в силу требуется около 15 минут. Однако иногда распространение внесенного изменения в системе DNS по всему Интернету занимает больше времени. Если после добавления записей DNS возникла проблема с потоком обработки почты или другие неполадки, см. статью [Устранение неполадок после смены имени домена или записей DNS](../get-help-with-domains/find-and-fix-issues.md). 
  
::: moniker-end

## <a name="more-about-updating-dns-records"></a>Дополнительные сведения об изменении записей DNS
<a name="BKMK_MoreAbout"> </a>

 **Если вы знаете, как обновлять записи DNS на узле DNS своего домена**, то используйте значения DNS Microsoft 365, чтобы отредактировать записи на узле DNS, например для настройки записи MX или SPF. Чтобы найти конкретные значения для использования, [выполните эти действия](../get-help-with-domains/information-for-dns-records.md). Также можно увидеть эти значения в мастере настройки доменов по мере выполнения действий в нем.
  
 **Если вам нужна помочь в добавлении необходимых записей DNS**, см. [Настройка домена (инструкции для конкретных узлов](https://docs.microsoft.com/office365/admin/get-help-with-domains/set-up-your-domain-host-specific-instructions?view=o365-worldwide). Затем выполните общие действия, указанные в этом разделе, чтобы создать записи DNS вашего домена для использования домена со службами Microsoft 365, такими как электронная почта.
  
 **Если с вашим личным доменом не используется никакой веб-сайт**, вы можете не выполнять настройку самостоятельно, а использовать Microsoft 365, чтобы настроить записи DNS вашего домена и управлять ими. Ознакомьтесь с [двумя вариантами настройки записей DNS для личного домена и управления ими](https://support.office.com/article/5980474a-097f-4f21-a864-21245314957f.aspx) в Microsoft 365.
