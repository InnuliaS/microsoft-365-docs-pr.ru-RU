# <a name="microsoft-365-for-enterprise-foundation-infrastructure-for-non-enterprises"></a>Базовая инфраструктура Microsoft 365 для предприятий для некорпоративных организаций

Некорпоративные организации также могут развернуть Microsoft 365 для предприятий, чтобы воспользоваться преимуществами интегрированной и безопасной инфраструктуры, удобной для совместной работы и развития творческого потенциала. Характерные особенности некорпоративных организаций:

- Локальная инфраструктура — небольшая (серверы электронной почты, файловые серверы и домен доменных служб Active Directory) или вовсе отсутствует.
- Небольшая численность ИТ-отдела, причем большинство ИТ-сотрудников занимаются задачами широкого профиля, а не специализируются на определенных задачах, таких как сети или электронная почта.

Для небольших некорпоративных организаций корпорация Майкрософт предлагает [Microsoft 365 для бизнеса](https://www.microsoft.com/microsoft-365/business). Однако существуют причины, по которым вам может потребоваться Microsoft 365 для предприятий, например:

- Вашей организации требуется или в будущем потребуется более 300 лицензий Microsoft 365, тогда как в Microsoft 365 для бизнеса это максимальное количество.
- Вашей организации требуются расширенные возможности эффективной работы, голосовой связи и аналитики, недоступные в Microsoft 365 для бизнеса.

В этой статье приведена упрощенная пошаговая инструкция по развертыванию базовой инфраструктуры Microsoft 365 для предприятий для вашей некорпоративной организации.

## <a name="first-set-up-your-subscription"></a>Во-первых, необходимо настроить подписку.

Нужно настроить домены системы доменных имен (DNS) для своей подписки. Если у вас уже есть подписка Microsoft 365, то это должно быть уже сделано. Если нет, то выполните инструкции, перечисленные в статье [Добавление домена в Office 365](https://docs.microsoft.com/office365/admin/setup/add-domain?view=o365-worldwide).

Затем нужно настроить дополнительные параметры безопасности для Microsoft 365. Выполните инструкции, перечисленные в статье [Настройка усиленной защиты](https://docs.microsoft.com/office365/securitycompliance/tenant-wide-setup-for-increased-security).

## <a name="phase-1-networking"></a>Этап 1. Сеть

В некорпоративных организациях обычно используется локальное подключение к Интернету, при этом не используются прокси-серверы, брандмауэры и устройства проверки пакетов. У поставщика услуг Интернета (ISP), обслуживающего каждый офис, имеется локальный для этого региона DNS-сервер, поэтому трафик направляется на сетевое расположение Microsoft 365, ближайшее к офисам и работающим в этих офисах локальным пользователям. Дополнительные сведения см. в статье [Настройка локальных подключений к Интернету для всех офисов](networking-dns-resolution-same-location.md).

Таким образом, необходимо убедиться у поставщика услуг Интернета, что подключение в каждом офисе:

- Использует локальный DNS-сервер для этого региона.
- Пригодно для существующих и будущих потребностей при увеличении объема используемых облачных служб Microsoft 365.

Если вы используете прокси-серверы, брандмауэры или средства проверки пакетов, сведения по оптимизации производительности служб Microsoft 365 см. в статье [Настройка обхода трафика](networking-configure-proxies-firewalls.md).

## <a name="phase-2-identity"></a>Этап 2. Идентификация

Каждый сотрудник вашей организации должен иметь возможность войти в систему. Для этого требуется учетная запись пользователя в клиенте Azure Active Directory (Azure AD) вашей подписки Microsoft 365 для предприятий. Для размещения учетных записей используются группы. Другие группы используются для общения и для получения доступа к разрешенным ресурсам, например к сайтам SharePoint Online или к команде. 

### <a name="administrator-accounts"></a>Учетные записи администратора

Для защиты учетных записей глобальных администраторов необходимо включить требование стойких паролей и многофакторной проверки подлинности (MFA). Дополнительные сведения см. в статье [Защита учетных записей глобальных администраторов](identity-create-protect-global-admins.md#protect-global-administrator-accounts).

Если в вашей организации требуется высокий уровень безопасности и вы используете Microsoft 365 E5, используйте службу Privileged Identity Management в Azure AD, чтобы включить предоставление своевременного доступа администраторам. Дополнительные сведения см. в статье [Настройка глобальных администраторов по требованию](identity-create-protect-global-admins.md#identity-pim).

### <a name="recommendations-for-groups"></a>Рекомендации для групп

Если у вас есть локальный домен AD DS, используйте эти группы в Microsoft 365 для предприятий в качестве групп в Azure AD.

Если у вас нет локального домена AD DS, создайте группы безопасности в Azure AD со следующими уровнями безопасности.

| Уровень безопасности | Описание | Примеры |
|:-------|:-----|:-----|
| Базовый | Это минимальный стандарт по умолчанию, используемый для защиты данных и удостоверений устройств, обращающихся к данным. <BR><BR> Этот уровень используется для большей части данных организации, находящихся под управлением большей части пользователей. | Группы для рядовых сотрудников, включая направления продаж, маркетинга, поддержки, администрирования и производства. |
| Конфиденциальный | На этом уровне обеспечивается дополнительная безопасность для данных, которым требуется более надежная защита по сравнению с базовым уровнем. В состав этих групп входят пользователи, которые используют и создают конфиденциальные данные, относящиеся к отделам и проектам, не предназначенным для всеобщего доступа. | Группы продуктов и маркетинга, занимающиеся разработкой будущих продуктов |
| Строго контролируемый | Это высший уровень защиты, обычно он требуется для относительно небольшого объема строго засекреченных данных, относящихся к интеллектуальной собственности или к коммерческим тайнам. Также этот уровень необходим для данным, для которых обязательно выполнение требований безопасности. |  Исследовательские, юридические и финансовые группы, а также группы, в которых хранятся или используются данные по клиентам или партнерам. |

### <a name="hybrid-identity"></a>Гибридная идентификация

Если у вас есть локальный домен AD DS, необходимо синхронизировать набор учетных записей пользователей, групп и контактов вашего домена с клиентом Azure AD вашей подписки Microsoft 365 для предприятий. Для ваших некорпоративных пользователей настройте Azure AD Connect на сервере с синхронизацией хэша паролей (PHS). Дополнительные сведения см. в разделе [Синхронизация удостоверений](identity-add-user-accounts.md#synchronize-identities-for-hybrid-identity).

### <a name="more-secure-user-access-with-conditional-access-policies"></a>Более безопасный доступ пользователей с помощью политик условного доступа

Azure AD проверяет условия входа пользователей и может задействовать политики условного доступа, чтобы предоставить или запретить доступ, а также требовать выполнения дальнейших действий для завершения входа. Например, если служба Azure AD определит, что вход происходит в условиях среднего или высокого риска, она может потребовать, чтобы пользователь прошел многофакторную проверку подлинности для завершения входа.

Политики условного доступа применяются к учетным записям пользователей или группам. Чтобы упростить назначение политик условного доступа, создайте в организации следующие группы безопасности AD:

- BASELINE

  Содержит группы и учетные записи пользователей с доступом к базовым данным.

- SENSITIVE

  Содержит группы и учетные записи пользователей с доступом к конфиденциальным данным.

- HIGHLY-REGULATED

  Содержит группы или учетные записи пользователей с доступом к строго контролируемым данным.

- COND-ACCESS-EXCLUDE

  Пустая группа, которую можно использовать, чтобы временно исключать пользователей из политик условного доступа.

Ниже приведен список политик условного доступа Azure AD, которые следует включить или создать.

| Политика условного доступа Azure AD | Группы, к которым применяется эта политика |
|:------|:-----|
| Базовая политика: требовать многофакторную проверку подлинности для администраторов | Эта политика применяется к ролям администраторов, поэтому нет необходимости создавать группы. Эту политику просто нужно включить. Все остальные политики необходимо создать и включить. |
| Блокировать клиенты, не поддерживающие современную проверку подлинности | В параметрах политики выберите "Все пользователи". |
| Требовать многофакторную проверку подлинности при среднем или высоком риске входа (требуется Microsoft 365 E5) | BASELINE |
| Требовать многофакторную проверку подлинности при среднем или высоком риске входа (требуется Microsoft 365 E5) | SENSITIVE |
| Всегда требовать многофакторную проверку подлинности | HIGHLY-REGULATED |
| Требовать использования одобренных приложений на устройствах с iOS и Android | BASELINE, SENSITIVE, HIGHLY-REGULATED |
| Требовать использования соответствующих политике компьютеров | BASELINE |
| Требовать использования соответствующих политике компьютеров, устройств с iOS и Android | SENSITIVE, HIGHLY-REGULATED |
|||

### <a name="groups-for-easier-management"></a>Упрощение управления с помощью групп

Ниже описаны некоторые функции, позволяющие упростить управление группами и лицензированием.

| Функция | Использование |
|:------|:-----|
| Самостоятельное управление группами | Разрешите управление группами Azure AD владельцам групп, а не ИТ-специалистам. Дополнительные сведения см. в статье [Самостоятельное управление группами](identity-use-group-management.md#allow-users-to-create-and-manage-their-own-groups). |
| Динамическое членство в группах | Настройте автоматическое добавление или удаление учетных записей пользователей из групп Azure AD на основании атрибутов учетных записей пользователей, таких как отдел или страна. Дополнительные сведения см. в статье [Динамическое членство в группах](identity-use-group-management.md#set-up-dynamic-group-membership). |
| Групповое лицензирование | Используйте членство в группах, чтобы автоматически назначать лицензии учетным записям пользователей или отменять назначение. Дополнительные сведения см. в статье [Групповое лицензирование](identity-use-group-management.md#set-up-automatic-licensing). |
|  |  |

При использовании группового лицензирования создайте группу с названием LICENSED. В этой группе будут содержаться имена учетных записей пользователей, которым назначается лицензия Microsoft 365 для предприятий.

### <a name="monitor-user-access"></a>Мониторинг доступа пользователей

Если вы используете Microsoft 365 E5, можно использовать защиту идентификации Azure AD для мониторинга и анализа попыток входа пользователей на предмет компрометации учетных данных. Дополнительные сведения см. в статье [Защита от компрометации учетных данных](identity-secure-user-sign-ins.md#protect-against-credential-compromise).

## <a name="phase-3-windows-10-enterprise"></a>Этап 3. Windows 10 Корпоративная

Чтобы убедиться, что устройства с Windows 10 Корпоративная интегрированы в инфраструктуру удостоверений и безопасности Microsoft 365 для предприятий, можно использовать следующие способы:

- Гибридная среда (есть локальный домен AD DS)

  Все существующие устройства с Windows 10 Корпоративная, уже присоединенные к вашему домену AD DS, необходимо подключить к клиенту Azure AD Инструкции см. в статье [Настройка гибридных устройств, присоединенных к Azure Active Directory](https://go.microsoft.com/fwlink/p/?linkid=872870).

  Все новые устройства с Windows 10 Корпоративная необходимо присоединить к домену AD DS, а затем присоединить к клиенту Azure AD.

  Все устройства с Windows 10 Корпоративная необходимо зарегистрировать в службе управления мобильными устройствами. Инструкции см. в статье [Регистрация устройств с Windows 10 в Intune с помощью групповой политики](https://go.microsoft.com/fwlink/p/?linkid=872871).

- Только облачная среда (нет локального домена AD DS)

  Присоедините каждое устройство с Windows 10 Корпоративная к клиенту Azure AD своей подписки.

  Дополнительные сведения см. в статье [Присоединение рабочего устройства к сети организации](https://docs.microsoft.com/azure/active-directory/user-help/user-help-join-device-on-network).


После установки и присоединения на все устройства с Windows 10 Корпоративная будут автоматически устанавливать устанавливаться обновления из облачной службы Центра обновления Windows для бизнеса. В некорпоративных организациях обычно не нужно настраивать инфраструктуру для распространения и установки обновлений Windows 10.

## <a name="phase-4-microsoft-365-apps-for-enterprise"></a>Этап 4. Приложения Microsoft 365 для предприятий

Приложения Microsoft 365 для предприятий, как и Office 2016 или Office 2019, устанавливается непосредственно на клиентских устройствах. При этом приложения Microsoft 365 для предприятий получают обновления, регулярно включающие новые функции. Дополнительные сведения см. в статье [Приложения Microsoft 365 для предприятий на предприятии](https://docs.microsoft.com/deployoffice/about-office-365-proplus-in-the-enterprise).

В некорпоративной организации необходимо вручную установить приложения Microsoft 365 для предприятий на устройствах, в том числе устройствах с Windows, iOS и Android. Это можно сделать при подготовке новых устройств к использованию, также это могут сделать пользователи в процессе перехода на новую инфраструктуру.

Как бы то ни было, администратор или пользователь входят на портал Office 365 в https://portal.office.com. На **главной вкладке Microsoft Office** щелкните **Установить Office** и пройдите все шаги процесса установки.

Обновления компонентов Microsoft 365 для предприятий ежемесячно устанавливаются на все компьютеры, на которых установлен этот пакет. В некорпоративных организациях обычно не нужно настраивать инфраструктуру для распространения и установки обновлений приложений Microsoft 365 для предприятий.

## <a name="phase-5-mobile-device-management"></a>Этап 5. Управление мобильными устройствами

Microsoft 365 для предприятий включает Microsoft Intune для управления мобильными устройствами. С помощью Intune можно управлять устройствами с Windows, iOS, Android и macOS, чтобы защитить ресурсы организации, включая данные. В Intune используются учетные записи пользователей, групп и компьютеров в Azure AD.

В Intune поддерживается два способа управления мобильными устройствами:

- Управление мобильными устройствами (MDM) при регистрации устройств в Intune. После регистрации эти устройства становятся управляемыми, они могут получать политики, правила и параметры, используемые в вашей организации. Как правило, такие устройства принадлежат вашей организации и выдаются сотрудникам.

- Пользователи со своими собственными личными устройствами могут не желать регистрировать эти устройства или разрешать управление своими устройствами системе Intune с вашими политиками и параметрами. Тем не менее, вам все равно нужно защитить ресурсы и данные организации. В этом случае вы можете защитить свои приложения с помощью управления мобильными приложениями (MAM).  

Политики Intune могут обеспечить соответствие устройств требованиям и защиту приложений. Вот список политик Intune, которые необходимо создать.

| Политика соответствия устройства требованиям для iOS | SENSITIVE, HIGHLY-REGULATED |
| Политика соответствия устройства требованиям для macOS | SENSITIVE, HIGHLY-REGULATED |
| Политика соответствия устройства требованиям для Android и Android Enterprise | SENSITIVE, HIGHLY-REGULATED |
| Политика защиты приложений для iOS | BASELINE, SENSITIVE, HIGHLY-REGULATED |
| Политика защиты приложений для macOS | BASELINE, SENSITIVE, HIGHLY-REGULATED |
| Политика защиты приложений для Android и Android Enterprise | BASELINE, SENSITIVE, HIGHLY-REGULATED |

Инструкции см. в статье [Основные политики идентификации и доступа для устройств](identity-access-policies.md).

## <a name="phase-6-information-protection"></a>Этап 6. Защита данных

В Microsoft 365 для предприятий предусмотрен целый ряд функций защиты информации, с помощью которых можно по-разному классифицировать данные, применяя к ним различные уровни управления, безопасности и защиты. 

Например, для обычной переписки большинства сотрудников и для документов, над которыми работают эти сотрудники, требуется некоторый базовый уровень защиты. Для финансовых записей, данных о клиентах и интеллектуальной собственности требуется более высокий уровень защиты.

Первый шаг в стратегии защиты информации состоит в определении уровней защиты. Во многих организациях используются следующие уровни, которые уже применяются для политик условного доступа:

- Базовый

  Примеры данных: обычная деловая переписка (электронная почта) и файлы администраторов, специалистов по продажам и сотрудников службы поддержки.

- Конфиденциальный

  Примеры данных: финансовые и юридические сведения, данные об исследованиях и разработке новых продуктов или услуг.

- Строго регулируемый уровень

  Примеры данных: личные сведения ваших клиентов и партнеров, интеллектуальная собственность или стратегические планы вашей организации.

На основе этих уровней безопасности необходимо определить и применить:

- Пользовательские типы конфиденциальной информации

  В Microsoft 365 предусмотрен широкий набор типов конфиденциальной информации, например, номера медицинского страхования или номера кредитных карт. Если в предоставленном списке нет нужного вам типа информации, можно создать собственный тип.

- Метки хранения

  Необходимо определить, как долго следует хранить документы определенных типов или с определенным содержимым в соответствии с правилами организации и региональными нормативными требованиями. Это задачу можно решить для электронной почты и документов с помощью меток хранения. Метки хранения также можно использовать совместно с политиками защиты от потери данных, которые могут ограничивать общий доступ к файлам или электронной почте за пределами организации.

- Метки конфиденциальности

  Можно помечать сообщения электронной почты и документы, используя именованные метки конфиденциальности, чтобы можно было применять дополнительные уровни безопасности. Это могут быть водяные знаки, шифрование и разрешения, указывающие, кому разрешен доступ к электронной почте или документу, и какие действия разрешены.

Дополнительные сведения см. в статье [Типы классификации Microsoft 365](infoprotect-configure-classification.md#microsoft-365-classification-types).

Если вы используете метки конфиденциальности с разрешениями, можно создать дополнительные группы безопасности, чтобы определить, каким пользователям доступны те или иные действия с электронной почтой и документами с примененными метками конфиденциальности. 

Например, нужно создать метку конфиденциальности RESEARCH, чтобы защитить электронную почту и документы исследовательской группы. Вы определяете, что:

- Исследователи должны иметь возможность изменения документов, помеченных меткой конфиденциальности RESEARCH.
- Сотрудники, не связанные с исследовательской деятельностью, должны иметь только возможность просмотра документов, помеченных меткой конфиденциальности RESEARCH. 

Это означает, что нужно создать еще две группы Microsoft 365 и управлять ими:

- RESEARCH-ALL
- RESEARCH-VIEW

Эти группы и их разрешения входит в состав конфигурации метки конфиденциальности RESEARCH.

Если метки конфиденциальности настроены с групповыми разрешениями, необходимо управлять членством в этих группах.

## <a name="onboarding"></a>Адаптация

Благодаря инфраструктуре Microsoft 365 для предприятий вы можете легко осуществить адаптацию новых сотрудников.

### <a name="a-new-windows-10-enterprise-device"></a>Новое устройство с Windows 10 Корпоративная

Перед выдачей сотруднику нового устройства с Windows 10 Корпоративная:

- Если используется гибридная идентификация

  Подключите устройство к домену AD DS, присоедините устройство к клиенту Azure AD, затем зарегистрируйте устройство в Intune.

- Если используется только облачная идентификация

  Подключите устройство к клиенту AD DS.

### <a name="existing-employee-with-an-ad-ds-user-account"></a>Существующий сотрудник с учетной записью пользователя AD DS

При первоначальном переходе вашей организации, если используется гибридная идентификация, добавьте учетную запись пользователя AD DS в следующие группы Azure AD:

- LICENSED
- Соответствующие группы безопасности AD DS или Azure AD, являющиеся участниками групп BASELINE, SENSITIVE и HIGHLY-REGULATED в Azure AD
- Группы меток конфиденциальности (по мере необходимости)

Существующий сотрудник уже должен быть уже добавлен в соответствующие группы AD DS на уровне рабочих групп, отделов и регионов.

В центре администрирования Microsoft 365 можно добавить учетную запись пользователя для нескольких групп Azure AD. В окне свойств учетной записи пользователя щелкните **Управление группами > Добавление членства в группах**.

Если вы хотите использовать оболочку PowerShell, см данную [загружаемую книгу Excel](https://github.com/MicrosoftDocs/microsoft-365-docs/raw/public/microsoft-365/media/deploy-foundation-infrastructure-non-enterprises/Group-License-Mgmt-PowerShell.xlsx), которая создает команды PowerShell на основе указанной учетной записи пользователя и выбранных имен групп.

### <a name="new-employee-with-a-cloud-only-user-account"></a>Новый сотрудник с полностью облачной учетной записью

При первоначальном переходе вашей организации, если используется только облачная идентификация, добавьте учетную запись нового пользователя в следующие группы:

- LICENSED
- Соответствующие группы безопасности Azure AD, являющиеся участниками групп BASELINE, SENSITIVE и HIGHLY-REGULATED в Azure AD
- Группы на уровне рабочих групп, отделов и регионов
- Группы меток конфиденциальности (по мере необходимости)

### <a name="initial-sign-in-to-microsoft-365"></a>Начальный вход в Microsoft 365

Когда сотрудники впервые входят в Microsoft 365, попросите их:

1. Войти на свои устройства с помощью учетных данных своих учетных записей пользователей.
2. В веб-браузере войти на портал Office 365 по адресу https://portal.office.com.
3. На **главной вкладке Office 365** щелкнуть **Установить Office**, чтобы установить приложения Microsoft 365 для предприятий на свои устройства.
