## <a name="recommended-client-configurations"></a>Рекомендуемые конфигурации клиента
В этом разделе рассматриваются рекомендуемые конфигурации клиента платформы по умолчанию, обеспечивающие оптимальный единый вход для пользователей, а также технические требования для условного доступа.

### <a name="windows-devices"></a>Устройства Windows
Рекомендуется Windows 10 (версия 1703 или более поздние), так как платформа Azure обеспечивает максимально удобные возможности единого входа в локальной среде и Azure AD. Выданные организацией или учебным заведением устройства должны быть настроены для прямого присоединения к Azure AD или, если организация использует локальное присоединение к домену AD, эти устройства должны быть [настроены для автоматической регистрации в Azure AD](https://docs.microsoft.com/azure/active-directory/active-directory-conditional-access-automatic-device-registration-setup).

При использовании устройств BYOD Windows пользователи могут нажать кнопку "Добавить рабочую или учебную учетную запись". Обратите внимание, что пользователи браузера Chrome в Windows 10 должны [установить расширение](https://chrome.google.com/webstore/detail/windows-10-accounts/ppnbnpeolgkicgegkbkbjmhlideopiji?utm_source=chrome-app-launcher-info-dialog), чтобы выполнять вход так же легко, как пользователи Edge или IE. Кроме того, если в организации есть устройства Windows 7, присоединенные к домену, можно установить Microsoft Workplace Join для компьютеров без Windows 10, [чтобы зарегистрировать](https://www.microsoft.com/download/details.aspx?id=53554) устройства в Azure AD.

### <a name="ios-devices"></a>Устройства iOS
Рекомендуется устанавливать [приложения Microsoft средства проверки подлинности](https://docs.microsoft.com/azure/multi-factor-authentication/end-user/microsoft-authenticator-app-how-to) на устройствах пользователей перед развертыванием условного доступа или политик многофакторной проверкой Подлинности. Как минимум должны быть установлены приложения, когда пользователям предлагается регистрация свои устройства с Azure AD, добавив рабочего или школе учетной записи или при установке приложения портала компании Intune возможность зачисления свои устройства в систему управления. Это зависит от политики условного доступа.

### <a name="android-devices"></a>Устройства Android
Перед развертыванием политик условного доступа или при необходимости во время определенных попыток проверки подлинности рекомендуется установить [приложение корпоративного портала Intune](https://play.google.com/store/apps/details?id=com.microsoft.windowsintune.companyportal&hl=en
) и [приложение Microsoft Authenticator](https://docs.microsoft.com/azure/multi-factor-authentication/end-user/microsoft-authenticator-app-how-to). После установки приложения пользователям может быть предложено зарегистрироваться в Azure AD и зарегистрировать устройства в Intune. Это зависит от политики условного доступа.

Также рекомендуется стандартизовать корпоративные устройства (COD) у изготовителей оборудования для версий, поддерживающих Android for Work или Samsung Knox, чтобы разрешить управление почтовыми учетными записями и их защиту с помощью политики Intune MDM.

## <a name="protecting-administrator-accounts"></a>Защита учетных записей администраторов
Azure Active Directory предоставляет простой способ начать защита доступа администратора с предварительно настроенным условного доступа политикой. В Azure AD перейдите к blade условного access и просмотрите элементы для данной политики — **базовой политики: требовать многофакторной проверкой Подлинности для администраторов**. Щелкните эту политику и выберите **использовать политику немедленно**. 

Эта политика требует многофакторной проверкой Подлинности для следующих ролей:
- Глобальный администратор
- Администраторы SharePoint
- Администраторы Exchange
- Администраторы условного доступа
- Администраторы безопасности

Для получения дополнительных сведений см [базовой политики безопасности для учетных записей администратора Azure AD](https://cloudblogs.microsoft.com/enterprisemobility/2018/06/22/baseline-security-policy-for-azure-ad-admin-accounts-in-public-preview/).

Дополнительные рекомендации относятся следующие:
- Используйте Azure AD привилегированной управления удостоверениями, чтобы уменьшить количество сохраняемых учетные записи администраторов. Ознакомьтесь [с загрузкой с помощью PIM](https://docs.microsoft.com/en-us/azure/active-directory/privileged-identity-management/pim-getting-started). 
- [Управление правами доступа в Office 365](https://docs.microsoft.com/en-us/office365/securitycompliance/privileged-access-management-overview) , чтобы защитить организацию от нарушения, которые могут использовать существующие привилегированный учетные записи администратора с положение доступа к конфиденциальным данным или доступ к параметрам конфигурации важных. 
- Используйте учетные записи администратора Office 365 только для администрирования. Администраторы должны иметь отдельного пользователя учетной записи для обычного использования без прав администратора и использовать только свои учетную запись администратора при необходимости для выполнения задач, связанных с их работы. Роли [администратора Office 365](https://support.office.com/article/About-Office-365-admin-roles-da585eea-f576-4f55-a1e0-87090b6aaa9d) имеют значительно больше прав доступа к службам Office 365.
- Следуйте рекомендациям по обеспечению безопасности привилегированных учетных записей в Azure AD, как описано в этой [статье](https://docs.microsoft.com/azure/active-directory/admin-roles-best-practices).
