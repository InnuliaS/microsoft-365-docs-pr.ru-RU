## <a name="recommended-client-configurations"></a>Рекомендуемые конфигурации клиента
В этом разделе описываются конфигурации клиента по умолчанию платформы, которую мы рекомендуем для обеспечения оптимального единого входа для пользователей, а также технические необходимые условия для условного доступа.

### <a name="windows-devices"></a>Устройства Windows
Мы рекомендуем Windows 10 (1703 или более поздняя версия), взаимодействия замыслу Azure для обеспечения наиболее единого входа для локальных и Azure AD. Рабочий или выдается школа устройств должны быть настроены для присоединения к Azure AD непосредственно или если в организации используются локальные AD присоединение к домену, эти устройства должен быть [автоматически настраиваются для и автоматическая регистрация с Azure AD](https://docs.microsoft.com/azure/active-directory/active-directory-conditional-access-automatic-device-registration-setup).

Для устройств BYOD Windows пользователи могут использовать **Добавить работу или школе учетной записи**. Обратите внимание, что браузер Chrome пользователям на Windows 10 необходимо [установить расширения](https://chrome.google.com/webstore/detail/windows-10-accounts/ppnbnpeolgkicgegkbkbjmhlideopiji?utm_source=chrome-app-launcher-info-dialog) для получения же легко интерфейс входа как пользователи с пограничного сервера/IE. Кроме того Если вашей организации есть присоединенный к домену устройств Windows 7, можно установить Microsoft присоединиться к рабочей области для компьютеров Windows 10, [Загрузите пакет для регистрации](https://www.microsoft.com/download/details.aspx?id=53554) устройства с Azure AD.

### <a name="ios-devices"></a>Устройства iOS
Рекомендуется устанавливать [приложения Microsoft средства проверки подлинности](https://docs.microsoft.com/azure/multi-factor-authentication/end-user/microsoft-authenticator-app-how-to) на устройствах пользователей перед развертыванием условного доступа или политик многофакторной проверкой Подлинности. Как минимум приложения должны быть установлены при пользователям предлагается регистрация свои устройства с Azure AD, добавив рабочего или школы учетной записи или при установке приложения портала компании Intune возможность зачисления свои устройства в систему управления. Это зависит от политики условного доступа.

### <a name="android-devices"></a>Устройства Android
Перед развертыванием политик условного доступа или при необходимости во время определенных попыток проверки подлинности рекомендуется установить [приложение корпоративного портала Intune](https://play.google.com/store/apps/details?id=com.microsoft.windowsintune.companyportal&hl=en) и [приложение Microsoft Authenticator](https://docs.microsoft.com/azure/multi-factor-authentication/end-user/microsoft-authenticator-app-how-to). После установки приложения пользователям может быть предложено зарегистрироваться в Azure AD и зарегистрировать устройства в Intune. Это зависит от политики условного доступа.

Мы также рекомендуем стандартизированы, что на изготовителей оборудования и версии, которые поддерживают Android на работе или Нокса Samsung разрешить почты учетных записей, управляемых и Intune MDM политики для защиты корпоративных устройств (COD).

## <a name="protecting-administrator-accounts"></a>Защита учетных записей администраторов
Azure AD предоставляет простой способ начать защита доступа администратора с политикой предварительно настроенного условного доступа. В Azure AD перейдите к **условного access** и просмотрите элементы для данной политики — **базовой политики: требовать многофакторной проверкой Подлинности для администраторов**. Выберите эту политику, а затем выберите **использовать политику немедленно**. 

Эта политика требует многофакторной проверкой Подлинности для следующих ролей:
- Глобальный администратор
- Администраторы SharePoint
- Администраторы Exchange
- Администраторы условного доступа
- Администраторы безопасности

Для получения дополнительных сведений см [базовой политики безопасности для учетных записей администратора Azure AD](https://cloudblogs.microsoft.com/enterprisemobility/2018/06/22/baseline-security-policy-for-azure-ad-admin-accounts-in-public-preview/).

Дополнительные рекомендации относятся следующие:
- Используйте Azure AD привилегированной управления удостоверениями, чтобы уменьшить количество сохраняемых учетные записи администраторов. Ознакомьтесь [с загрузкой с помощью PIM](https://docs.microsoft.com/en-us/azure/active-directory/privileged-identity-management/pim-getting-started). 
- [Управление правами доступа в Office 365](https://docs.microsoft.com/en-us/office365/securitycompliance/privileged-access-management-overview) , чтобы защитить организацию от нарушения, которые могут использовать существующие привилегированный учетные записи администратора с положение доступа к конфиденциальным данным или доступ к параметрам конфигурации важных. 
- Используйте учетные записи администратора Office 365 только для администрирования. Администраторы должны иметь отдельного пользователя учетной записи для обычного использования без прав администратора и использовать только свои учетную запись администратора при необходимости для выполнения задач, связанных с их работы. Роли [администратора Office 365](https://support.office.com/article/About-Office-365-admin-roles-da585eea-f576-4f55-a1e0-87090b6aaa9d) имеют значительно больше привилегий, чем служб Office 365.
- Следуйте рекомендациям по обеспечению безопасности привилегированных учетных записей в Azure AD, как описано в этой [статье](https://docs.microsoft.com/azure/active-directory/admin-roles-best-practices).
